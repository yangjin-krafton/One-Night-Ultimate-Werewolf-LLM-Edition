# 한밤의 늑대인간 LLM 에디션  
사내 웹/모바일 보드게임 기획서 (최종본)

---

## 1. 프로젝트 개요

### 1.1 프로젝트명
- **프로젝트명**: 한밤의 늑대인간 LLM 에디션
- **영문명**: One Night Ultimate Werewolf – LLM Edition
- **용도**: 사내용 파티게임 + LLM/실시간 웹/TTS 데모 프로젝트

### 1.2 목표
- 사내망 환경에서 **로컬 PC 한 대에 서버**를 띄우고,
- 구성원들이 **각자 스마트폰(웹 브라우저)** 으로 접속해 즐기는 디지털 버전의 *한밤의 늑대인간* 구현.
- 기존 오프라인 진행 + 공식 앱 사회자를 대체/강화:
  - **역할 랜덤 배정 & 카드 조작/교환 로직** 자동화
  - **밤/낮/투표 단계 관리 및 타이머** 자동 제어
  - **방장(호스트) 휴대폰에서 TTS 안내 방송** 재생
- LLM은:
  - 기획/코딩/데이터/스토리 대사 생성에 적극 활용하되,
  - 실제 런타임 의존도는 낮게 (오프라인 스토리 스크립트 중심).

## 2. 타깃 환경 및 제약

### 2.1 서버 환경
- **위치**: 회사 내부망에 연결된 로컬 PC (Windows 또는 Linux)
- **역할**:
  - 방 생성/삭제/목록 관리
  - 게임 상태 관리 (카드/플레이어/단계)
  - WebSocket 기반 실시간 이벤트 전달
  - (선택) LLM 및 스토리 데이터 제공

### 2.2 클라이언트 환경
- **디바이스**: 각 플레이어의 **스마트폰**
  - iOS Safari / Chrome
  - Android Chrome
- **접속 방식**:
  - `http://<서버 IP 또는 호스트명>:포트` 접속
  - PWA(추가 설치 없이 홈 화면 추가) 지원 가능하면 좋음
- **특징**:
  - 별도 로그인/계정 없이 **닉네임만** 사용
  - 야간에는 **소리 없는 UI** (알림/벨소리 금지, 필요 시 진동 정도만)

### 2.3 네트워크/보안
- **사내망 전용 서비스**
- 외부 인터넷 연결 없이도 동작 가능하도록 설계
  - (LLM/TTS 외부 API는 옵션)
- 게임 데이터(닉네임/역할 등)는 **메모리 중심**, 필요 시 단기 로그만 파일로 저장

## 3. 룰 및 게임 디자인

### 3.1 기본 룰 (원작 기준 요약)
- 플레이어 수: **3~10명** (권장 4~8명)
- 사용 역할 카드: 플레이어 수 + 3장 (3장은 중앙 카드)
- 진행 구조:
  1. **밤 1회**: 각 역할이 순서대로 눈을 뜨고 능력 사용
  2. **낮 1회**: 토론 후 동시에 투표 → 처형 대상 결정
- 승리 조건:
  - **마을팀(Villagers)**:
    - 실제 늑대인간이 있는 경우: 늑대인간 1명 이상 처형 시 승리
    - 실제 늑대인간이 없는 경우: 아무도 죽지 않으면 승리
  - **늑대팀(Werewolves)**:
    - 늑대인간이 1명 이상 존재하고,
    - 처형된 사람 중에 늑대인간이 없으면 승리
  - **특수 역할(Tanner 등)**: 개별 승리 조건 따름

### 3.2 1차 버전 대상 역할
- **Villager(마을 주민)** – 능력 없음
- **Werewolf(늑대인간)** – 밤에 서로 존재 확인
- **Seer(예언자)** – 플레이어 1명 카드 OR 중앙 카드 2장 확인
- **Robber(도둑)** – 남의 카드와 자기 카드 교환 후 새 카드 확인
- **Troublemaker(말썽꾼)** – 남 둘의 카드 위치 교환 (본인은 결과 모름)
- **Drunk(주정뱅이)** – 자기 카드와 중앙 카드 1장 교환 (결과 모름)
- **Hunter(헌터)** – 헌터가 죽으면, 헌터가 지목한 플레이어도 같이 죽음
- **Tanner(탄너)** – 자신이 죽으면 본인만 승리
- **Mason(장인)** x2 – 밤에 서로의 정체를 알고 있는 마을팀

→ 이후 확장팩(데이브레이크 등) 역할은 차기 버전에서 추가.

## 4. 핵심 UX/플로우 설계

### 4.1 메인 진입 화면
모든 디바이스(PC/휴대폰) 공통:

- 버튼:
  - `방 만들기 (호스트로 시작)`
  - `방 목록 보기 (참여)`

### 4.2 방 생성 (호스트)

#### 4.2.1 생성 위치
- **PC**/모바일 어디서든 호스트가 될 수 있음.
- “방 만들기” 선택 시:

#### 4.2.2 방 생성 플로우
1. 닉네임 입력 (예: “jin”, “OO팀장”)
2. 방 옵션 설정:
   - 방 이름 (예: “점심 한밤방”)
   - 최대 인원 (3~10)
   - 역할 프리셋 선택:
     - 플레이어 수에 따라 추천 프리셋 제공
     - “커스텀”에서 세부 역할 on/off 가능
   - TTS 사용 여부 / 속도 / 목소리 (기본: 시스템 default)
3. 방 생성 완료 화면:
   - 방 코드 (숫자 4~6자리)
   - 접속 URL
   - QR 코드

호스트는 자동으로 **호스트 대기실 화면**으로 이동.

### 4.3 방 목록에서 참여

1. “방 목록 보기” 선택 시:
   - 서버에서 **현재 대기 중 방 리스트** 조회:
     - 방 이름
     - 호스트 닉네임
     - 현재 인원 / 최대 인원
     - 상태 (대기 / 게임 중)
2. 참여자는 방 하나 선택 → 닉네임 입력 → 입장

※ 방 코드/QR은 “직접 링크 전달” 시 편의를 위해 제공하되,  
기본 진입은 “방 목록 선택” 구조.

### 4.4 대기실(로비)

- 공통:
  - 현재 참가자 리스트 (닉네임)
  - 선택된 역할 리스트 (아이콘 + 간단 설명)
- **호스트 UI**:
  - 역할 구성 수정
  - TTS 옵션 수정
  - “게임 시작” 버튼
- **참여자 UI**:
  - 내 닉네임 확인
  - “호스트가 게임 시작을 누를 때까지 기다려주세요” 메시지

### 4.5 내 역할 확인 단계

- 게임 시작 후, 서버에서 역할 랜덤 배정:
  - 플레이어 수 + 3장(중앙 카드 3장)
- 각 플레이어 휴대폰:
  - 큰 카드 모양으로 **자기 역할 1장** 표시
  - 간단한 능력 설명
  - UI:
    - “길게 누르면 역할 보기” 방식 (옆 사람이 보기 어렵게)
    - 일정 시간 후 자동 블러 처리 가능(옵션)

### 4.6 밤 진행 – 핵심 컨셉

> **밤의 음성 안내는 오직 “호스트 휴대폰 TTS”에서만 나온다.**  
> 각 플레이어 폰은 조용히, 필요할 때만 카드 선택 화면이 잠깐 뜬다.

#### 4.6.1 밤 단계 공통 흐름

1. 호스트 TTS:
   - “지금부터 밤이 시작됩니다. 모두 눈을 감아주세요.”
2. 서버: `phase = NIGHT` 및 세부 sub-phase 단계로 진행
3. 역할 순서대로:
   - 방장 TTS:
     - “지금, 예언자는 눈을 뜹니다.”
     - (2초 정적)
     - “예언자는 확인할 대상을 조용히 선택하세요.”
   - 예언자 역할을 가진 플레이어에게만 휴대폰 화면에 **선택 UI** 표시
   - 예언자는 휴대폰에서 카드/플레이어 선택 → 서버에 전달
   - 서버가 결과 계산 후, 예언자에게만 결과 표시
   - 예언자는 조용히 다시 눈을 감음
4. 다음 역할로 반복 (늑대인간, 도둑, 말썽꾼, …)

> 플레이어 휴대폰에서는 **알림음/보이스 없음**,  
> 필요할 때만 화면에 조용히 카드 선택 창이 떠서 조작 후 바로 사라지는 구조.

### 4.7 낮 진행

1. 호스트 TTS:
   - “아침이 밝았습니다. 모두 눈을 뜨세요.”
2. 모든 플레이어 휴대폰:
   - “낮입니다. 자유롭게 토론하세요.”
   - 상단에 **토론 남은 시간 타이머** 표시
3. 옵션:
   - 각자 메모 기능 (로컬 전용, 서버 전송 X)
   - (선택) 텍스트 채팅 기능 – 1차 버전에서는 생략 가능

### 4.8 투표 단계

1. 호스트 TTS:
   - “이제 누구를 처형할지 결정해야 합니다. 모두 동시에 투표를 준비하세요.”
2. 플레이어 폰:
   - 플레이어 리스트 카드 형태
   - 한 명 선택 후 “투표 확정”
3. 서버:
   - 모든 투표 집계
   - 최다 득표자 계산 (동률 처리 로직 포함)

### 4.9 결과 발표

1. 호스트 TTS:
   - 테마에 맞는 연출 멘트 재생
     - 예: “갈대숲이 술렁이고, 한 사람이 쓰러집니다…”
2. 모든 플레이어 폰:
   - 승리 팀 표시 (마을팀/늑대팀/탄너 등)
   - 각 플레이어:
     - 최종 역할
     - (옵션) 밤 동안 카드 이동 타임라인 간단 표시
3. “다시 하기” 버튼:
   - 같은 인원/방 유지, 역할만 다시 셔플해서 새 판 진행

## 5. 카드/역할 데이터 모델 및 능력 사용 로직

### 5.1 카드 데이터는 “서버에만” 존재

- 서버:
  ```ts
  type RoleId = 'villager' | 'werewolf' | 'seer' | 'robber' | 'troublemaker' | 'drunk' | 'hunter' | 'tanner' | 'mason';

  interface GameState {
    roomId: string;
    players: Player[];            // playerId, nickname 등
    playerRoleMap: { [playerId: string]: RoleId }; // 현재 각 플레이어의 카드
    centerCards: RoleId[];        // 중앙 카드 3장
    phase: GamePhase;
    // 기타 상태 (로그, 야간 순서 등)
  }

* 클라이언트(플레이어 휴대폰)는 **자기가 볼 수 있는 정보만** 서버가 잘라서 보내줌:

  * 내 현재 역할
  * 내가 야간에 확인한 결과 (예언자/늑대/장인 등)
  * 게임 단계 및 타이머

### 5.2 예언자 (Seer)

* 예언자 턴:

  1. 서버: `phase = NIGHT_SEER_TURN`
  2. 예언자 폰에:

     * “플레이어 1명 확인” 또는 “중앙 카드 2장 확인” UI 표시
  3. 예언자 선택 → 서버에 액션 전송
  4. 서버:

     * playerRoleMap / centerCards에서 해당 정보 조회
     * 예언자에게만 결과 전송
  5. 예언자 폰:

     * 길게 누르면 보이는 카드 UI 등으로 잠깐 표시
     * 일정 시간 후 자동 숨김

### 5.3 도둑 (Robber)

1. 도둑 턴:

   * 도둑 폰: “카드를 훔칠 플레이어 1명을 선택”
2. 선택 → 서버에 액션 전송
3. 서버:

   * `playerRoleMap[robber]` 와 `playerRoleMap[target]` 교환
   * 도둑에게만 **새 역할** 전송
4. 도둑 폰:

   * “이제 당신의 카드는 XXX입니다.” 표시

### 5.4 말썽꾼 (Troublemaker)

1. 말썽꾼 턴:

   * 말썽꾼 폰: “카드를 바꿀 플레이어 2명을 선택 (본인 제외)”
2. 서버:

   * 두 플레이어의 `playerRoleMap` 값만 교환
   * 말썽꾼에게도 결과는 알려주지 않음

### 5.5 주정뱅이 (Drunk)

1. 주정뱅이 턴:

   * “중앙 카드 중 하나를 선택”
2. 서버:

   * `playerRoleMap[drunk]`와 `centerCards[i]` 교환
   * 주정뱅이에게는 결과 정보 제공 X

### 5.6 늑대인간 / 장인 / 기타

* 늑대인간:

  * 밤에 서로의 존재를 확인
  * 외로운 늑대 옵션: 중앙 카드 1장 확인
* 장인:

  * 서로의 존재(닉네임)만 표시
* 헌터/탄너:

  * 낮/결과 단계에서 승리 판정에만 사용

## 6. LLM 활용 계획

### 6.1 개발 단계 활용

* **코딩 보조**:

  * TypeScript/React 컴포넌트 초안
  * 서버 상태머신 pseudo-code → 실제 코드 변환
  * 역할/스토리 데이터 구조 설계
* **UI/텍스트**:

  * 버튼/메시지 카피 문구
  * 튜토리얼/도움말 텍스트 초안

### 6.2 스토리/대사 생성 (오프라인)

#### 6.2.1 테마 기반 스토리보드

* 테마 예시:

  * `gothic_horror`: 고딕 호러풍
  * `comic_village`: 가벼운 개그풍
  * `occult_church`: 의식/종교적 분위기
* 각 테마마다:

  * 밤 시작 멘트 N개
  * 각 역할 등장 멘트 N개
  * 낮 시작/투표/결과 멘트 N개
* LLM 프롬프트로 한 번에 JSON 생성:

  * `segments[]` (텍스트 + pauseMs) 구조로
* 생성된 JSON 파일:

  * `stories_gothic.json`, `stories_comic.json` 등으로 서버에 저장
  * 게임 시작 시 하나의 테마를 선택 → 각 단계에서 랜덤으로 사용

#### 6.2.2 런타임 의존 최소화

* 게임 중에는 LLM 호출 없이:

  * **미리 생성해 둔 스토리 스크립트**만 사용
* 옵션 기능으로만:

  * “토론 요약”, “오늘의 MVP/수상한 자 분석” 등 LLM 기능 추가 가능

## 7. TTS 설계 (방장 휴대폰 기준)

### 7.1 기본 방향

* **야간/낮/투표/결과 안내 음성은 “방장(호스트) 휴대폰”에서만 재생**
* 참여자 폰은 기본적으로 음성 출력 없음 (조용히 카드 조작만)
* 장점:

  * “누가 어떤 역할인지” 휴대폰 알림/소리로 들키지 않음
  * 실제 보드게임처럼 방장 목소리만 들리는 구조를 재현

### 7.2 구현 전략 (1차 버전)

1. **웹 브라우저(Web Speech API)** 사용

   * 방장 휴대폰에서:

     * `window.speechSynthesis` 를 사용하여 TTS 재생
     * `lang = "ko-KR"` 설정
   * 주의:

     * 모바일 브라우저는 자동 재생 제한 → 게임 시작 전 “음성 테스트” 버튼 한 번 누르게 해서 허용

2. **스크립트 단위 재생**

   * JSON 스크립트:

     * `[{ text, pauseMs }, ...]` 형태
   * 재생 로직:

     * 각 segment마다 TTS 호출 → pauseMs 만큼 대기

### 7.3 향후 확장 전략

* 필요 시:

  * Capacitor/Cordova 기반 하이브리드 앱으로 감싸서
  * iOS/Android 네이티브 TTS API 직접 사용
* 매우 고급 옵션:

  * 온디바이스 경량 한국어 TTS 모델(WebAssembly 등) 적용 실험
  * 1차 버전의 필수 범위는 아님

## 8. 시스템 아키텍처 개요

### 8.1 구성

* **웹 서버 (게임 서버)**

  * Node.js + TypeScript (Express/Fastify + Socket.io) 권장
  * REST:

    * 방 생성/목록/입장
  * WebSocket:

    * 게임 상태 이벤트 브로드캐스트
    * 플레이어 액션 수신
* **클라이언트**

  * React + Vite (SPA)
  * 상태 관리 (예: Zustand/Redux)
  * 모바일 최적화 레이아웃 + 다크 테마
* **스토리/LLM 데이터**

  * 서버 로컬 JSON 파일로 관리
  * 필요 시 LLM API로 업데이트/생성

### 8.2 네트워크 설계 포인트

* **최소 통신만**:

  * 서버 → 클라이언트:

    * 현재 phase/타이머
    * 내가 행동 가능한지 여부
    * 내가 확인한 카드 결과
  * 클라이언트 → 서버:

    * 야간 역할 행동 (선택 대상)
    * 낮 투표 결과
* 타이머 동기화:

  * 서버에서 “종료 시각(timestamp)”만 전송
  * 각 클라이언트는 로컬 시간 기준으로 카운트다운

## 10. 요약

* **카드/역할 데이터는 서버에서만 진짜로 존재**하고,
* 각 플레이어 휴대폰은 **일시적인 “비밀 보는 창문 + 조작 패드”** 역할만 한다.
* **밤에는 방장 폰 TTS만 말하고**, 참여자 폰은 조용히 카드 선택/확인 UI만 잠깐 띄운다.
* **LLM은 기획/코딩/스토리 생성에 적극 활용하되**,
  런타임에서는 미리 생성된 스토리 대사를 사용하는 구조로 네트워크/안정성을 확보한다.
* 사내 환경에서 가볍게 돌릴 수 있으면서도,
  실제로 해보면 꽤 “몰입감 있는 파티 게임 + AI 데모”로 활용 가능한 구성을 목표로 한다.

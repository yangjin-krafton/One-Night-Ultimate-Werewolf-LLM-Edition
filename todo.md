  - [ ] NIGHT: 화면 전체를 어둡게(“눈 감기”) + 내 차례일 때만 하단에 행동 패널
  - [ ] DEBATE: 토론 안내 + 남은 시간(있다면) + 투표 대기 상태
  - [ ] VOTE: 큰 버튼(좌석 선택) + “제출됨” 확인
  - [ ] RESULT: 결과 요약(공개 정보) + 다음 단계 안내
  - [ ] REVEAL: 터치/스와이프 카드 뒤집기(내 카드 오픈)
  - [ ] ENDING: 엔딩 텍스트(가독성) + 다음 라운드 준비 CTA
- [ ] 모달/오버레이 UX
  - [ ] 시나리오 선택 모달(호스트): 검색/태그/권장 인원 뱃지
  - [ ] 오디오 권한(호스트): “오디오 활성화” 1회 클릭 유도(정책 대응)
  - [ ] 오류 토스트: 연결 끊김/재연결/오디오 재생 실패
- [ ] SVG 아이콘/일러스트(간단 자산 계획)
  - [ ] 상단 상태: Wi‑Fi/연결 점, 호스트 왕관, 스피커(나레이션), 음표(BGM)
  - [ ] 페이즈 아이콘: 달(NIGHT), 말풍선(DEBATE), 체크(VOTE), 메달/표(RESULT), 눈(REVEAL)
  - [ ] 카드 패턴: 카드 배경에 아주 약한 노이즈/그레인 SVG(선택)
- [ ] 모션(애니메이션) 상세 설계
  - [ ] 좌석 재정렬: 카드가 “앞 칸으로 슬라이드” (200~280ms, ease-out)
  - [ ] 페이즈 전환: 상단 바 색/아이콘이 스냅 + 본문은 페이드(180ms)
  - [ ] “내 차례” 강조: 카드 글로우 펄스(1.2s, 2회 후 정지)
  - [ ] 투표 제출: 버튼 축소→체크 표시(160ms)
  - [ ] REVEAL 카드 뒤집기: 3D flip(320ms) + 햅틱(가능 시) / 진동(옵션)
  - [ ] BGM fade: 800ms fade-in / 1200ms fade-out (기본값, 시나리오에서 override 가능)
  - [ ] 접근성: `prefers-reduced-motion`이면 모든 모션 최소화

- [ ] 그라데이션 “무드”를 게임 전체에 적극 반영(핵심 비주얼 방향)
  - [ ] 전역 배경: **페이즈에 따라** 배경 그라데이션이 천천히 움직임(과하지 않게)
    - [ ] 방법: `background-size: 200% 200%` + `@keyframes gradient-pan { 0%{background-position:0% 50%} 50%{background-position:100% 50%} 100%{background-position:0% 50%} }`
    - [ ] 속도: 12~24s(페이즈별로 다르게) / easing: linear
    - [ ] 레이어: 2~3겹(기본 그라데이션 + radial highlight + 매우 약한 노이즈)
  - [ ] 페이즈별 추천 그라데이션(예시 톤)
    - [ ] NIGHT: 남색/보라 + radial vignette(시야 좁힘)
    - [ ] DEBATE: 차분한 청록/블루(중립)
    - [ ] VOTE: 앰버/레드 포인트(긴장)
    - [ ] RESULT: 라임/시안(결과 강조)
    - [ ] REVEAL: 보라/핑크(드라마틱) + card flip 시 빛 번짐
    - [ ] ENDING: 시나리오 분위기 태그에 따라 프리셋 매핑(공포=저채도, 코믹=고채도)
  - [ ] 카드 “상태”를 그라데이션 모션으로 표현
    - [ ] 내 차례: 카드 테두리 그라데이션이 2회 펄스(느린 shimmer)
    - [ ] 투표 완료: 카드 하단에 얇은 그라데이션 바(고정) + 체크 아이콘
    - [ ] 연결 끊김: desaturate + 대각선 스트라이프(반투명) 오버레이
  - [ ] 버튼/CTA: 그라데이션 + 광택(shine) 모션(짧고 절제)
    - [ ] 기본 CTA: `linear-gradient(135deg, ... )` + `filter: drop-shadow(...)`
    - [ ] hover/press: 그라데이션 각도/위치 변경 + 살짝 눌림(translateY 1~2px)
  - [ ] 성능/발열 고려(모바일)
    - [ ] 그라데이션 모션은 `opacity/transform/background-position` 중심(레이아웃 변경 금지)
    - [ ] 저사양 대비: `prefers-reduced-motion` 또는 “절전 모드 토글”로 애니메이션 정지
    - [ ] 애니메이션은 동시에 1~2개만(배경 1 + 강조 1)로 제한

### 5.1 Windows 개발 중 “멀티 유저” 테스트(가상 페이지 여러 개)

> 목표: Windows PC 한 대에서 브라우저 탭/창을 여러 개 띄워도, 각 페이지가 서로 다른 유저로 인식되어 테스트 가능.

- [ ] 클라이언트 식별자(`clientId`) 정책
  - [ ] 기본: 첫 로드 시 `sessionStorage.clientId`가 없으면 랜덤 UUID 생성(탭/창마다 분리)
  - [ ] 새로고침/뒤로가기에도 동일 탭에서는 유지
- [ ] 디버그용 URL 파라미터 지원(테스트 편의)
  - [ ] `?debugClientId=U1&debugName=Alice` 형태로 강제 지정 가능
  - [ ] 파라미터가 있으면 `sessionStorage`를 덮어써서 확실히 분리
- [ ] Socket.io 핸드셰이크에 `clientId`를 포함
  - [ ] `socket.auth = { clientId, name }` 또는 query로 전달
  - [ ] 서버는 `clientId` 기준으로 “같은 유저 재접속”을 식별(옵션), 또는 항상 신규로 처리(MVP)
- [ ] 테스트 런처 페이지(권장)
  - [ ] `public/dev/multi.html`에서 “N명 열기” 버튼 → `window.open('/?debugClientId=U1&debugName=U1')` 식으로 여러 창을 자동 생성
  - [ ] 각 창은 고유 `debugClientId`로 접속 → 좌석/호스트 승계/투표 테스트가 쉬워짐
- [ ] 운영/실사용 안전장치
  - [ ] `debug*` 파라미터는 `NODE_ENV=development`에서만 허용(또는 숨김 토글)

### 5.2 Chrome 콘솔 디버그 커맨드(개발 편의)

> 목표: 개발 중 크롬 콘솔에서 명령어로 시나리오 언락/즉시 종료/승리 강제/페이즈 점프 등을 빠르게 테스트.

- [ ] 클라이언트에 전역 디버그 API 노출(개발 모드 한정)
  - [ ] `window.__gameDebug` 또는 `window.gameDebug` 객체
  - [ ] 타입/자동완성(JSDoc 또는 TS d.ts) 제공(선택)
- [ ] 기본 커맨드 목록(안)
  - [ ] 시나리오
    - [ ] `gameDebug.listScenarios()` : 시나리오 목록 출력
    - [ ] `gameDebug.selectScenario(scenarioId)` : 호스트로 시나리오 선택 요청
    - [ ] `gameDebug.unlockScenario(scenarioId)` : 잠금/필터 무시하고 강제 선택 가능하게(개발용)
  - [ ] 게임 진행
    - [ ] `gameDebug.startGame()` : 호스트 게임 시작 트리거
    - [ ] `gameDebug.endGame({ winner: "village" | "wolf" | "mixed" })` : 즉시 종료 + 엔딩 분기 강제
    - [ ] `gameDebug.setPhase(phaseName, { durationSec })` : 페이즈 강제 전환/타이머 재설정
    - [ ] `gameDebug.skipToVote()` / `gameDebug.skipToReveal()` : 자주 쓰는 단축
  - [ ] 투표/액션
    - [ ] `gameDebug.forceVotes({ seat1: seatN, ... })` : 투표 결과 강제(서버 권위 모델에서만)
    - [ ] `gameDebug.forceAction({ seat, action, target })` : 밤 행동 강제(선택)
  - [ ] 오디오
    - [ ] `gameDebug.playNarrationClip(clipId)` : 특정 클립 재생 테스트
    - [ ] `gameDebug.setBgm({ on, volume, fadeInMs, fadeOutMs })` : BGM 강제 제어
  - [ ] 연결/상태
    - [ ] `gameDebug.dumpState()` : 마지막 `room_snapshot` 출력
    - [ ] `gameDebug.reconnect()` / `gameDebug.disconnect()` : 소켓 재연결 테스트
- [ ] 구현 방식(권장)
  - [ ] 클라이언트 디버그 API는 내부적으로 **Socket.io 이벤트**를 호출
  - [ ] 서버에 `debug:*` 네임스페이스 이벤트 추가(예: `debug:set_phase`, `debug:end_game`, `debug:unlock_scenario`)
  - [ ] 서버는 `NODE_ENV=development` 또는 `DEBUG_COMMANDS=1`일 때만 처리
- [ ] 실수 방지(중요)
  - [ ] 사내 운영 환경에서는 디버그 이벤트 자체를 등록하지 않음(또는 토큰 필요)
  - [ ] 디버그 이벤트가 호출되면 서버 로그에 “DEBUG”로 명확히 남김
